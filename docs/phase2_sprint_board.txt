Phase 2 Sprint Task Board (Low-Cost / Minimal Subscription)

Guardrails
- No mandatory paid SaaS in Phase 2.
- Keep core capabilities self-hosted: FastAPI + PostgreSQL + frontend + optional Redis.
- Use free/public data sources where needed (for example ECB exchange rates).
- Prefer in-app notifications first; optional email uses existing SMTP mode.

Status Legend
- TODO: not started
- IN_PROGRESS: implementation started
- DONE: merged locally and validated with tests/checks

Sprint 1 (Tags + Budget Alerts, 1-2 weeks)

Ticket: P2-S1-T1
Title: Tags Schema (Expense Service)
Status: DONE
Type: DB
DB Tasks:
- Add table expenses_db.tag:
  tag_id (uuid pk), user_id (int), name (varchar 64), slug (varchar 80), created_at, updated_at.
- Add table expenses_db.expense_tag:
  expense_id (uuid fk), tag_id (uuid fk), created_at.
- Add constraints/indexes:
  unique (user_id, lower(name)),
  unique (user_id, slug),
  index expense_tag(expense_id),
  index expense_tag(tag_id).
Acceptance Criteria:
- Migration is idempotent.
- Duplicate tag names per user are rejected.

Ticket: P2-S1-T2
Title: Tags API + Expense Tag Filtering
Status: DONE
Type: API
API Tasks:
- Add endpoints:
  GET /api/v1/tags
  POST /api/v1/tags
  DELETE /api/v1/tags/{tag_id}
- Extend expense APIs:
  POST/PATCH /api/v1/expenses accepts tag_ids or tags[].
  GET /api/v1/expenses supports filter by tag_id and tag slug.
Acceptance Criteria:
- Users can CRUD only their own tags.
- Expense list filter returns only matching tagged expenses.

Ticket: P2-S1-T3
Title: Budget Alert Schema (Budget Service)
Status: DONE
Type: DB
DB Tasks:
- Add budgets_db.budget_alert_config:
  config_id (uuid pk), user_id, budget_id, threshold_percent (numeric), channel (in_app|email), is_active, created_at, updated_at.
- Add budgets_db.budget_alert_event:
  event_id (uuid pk), user_id, budget_id, period_start, period_end, threshold_percent, spent_amount, budget_amount, channel, sent_at.
- Add unique guard:
  unique(user_id, budget_id, period_start, period_end, threshold_percent, channel).
Acceptance Criteria:
- Same threshold event is recorded only once per budget period.
- Migration is idempotent.

Ticket: P2-S1-T4
Title: Budget Alert Evaluation API/Job
Status: DONE
Type: API/Worker
API Tasks:
- Add budget service evaluation command:
  python -m app.jobs.budget_alert_processor --as-of YYYY-MM-DD
- Evaluation flow:
  load active budgets + alert config,
  fetch spend totals (expense summary endpoint),
  create budget_alert_event when threshold crossed and unsent for period.
- Optional endpoint:
  POST /api/v1/budgets/alerts/evaluate (admin/internal use).
Acceptance Criteria:
- Running processor twice does not duplicate alert events for same period.
- Logs include processed_count, sent_count, skipped_count, failed_count.
